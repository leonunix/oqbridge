OpenSearch + Quickwit / Hot-Cold Tiering Proxy

让热数据留在 OpenSearch，让冷数据安静地去 Quickwit。

oqbridge 是一个面向日志与时序检索场景的轻量级桥接层。 它把用户查询入口保持在 OpenSearch 兼容接口之上，同时把历史数据自动迁移到 Quickwit， 以更低的存储成本承接更长周期的数据保留需求。

Project Overview

一个桥，而不是另一个孤立系统

解决的问题

许多团队希望把最近的活跃数据保留在 OpenSearch 里，以获得成熟的检索与生态支持， 但又不愿意为数月甚至数年的历史数据持续支付高昂存储成本。

oqbridge 的思路不是替换已有平台，而是在现有查询入口前增加一层足够轻、足够透明的桥接能力， 让冷热分层成为基础设施细节，而不是业务负担。

设计原则

  • 保留 OpenSearch 兼容接口，降低改造成本
  • 让冷数据迁移具备断点续传与多实例安全
  • 用清晰的时间边界驱动路由，而不是复杂手工规则
  • 把用户认证继续放在 OpenSearch，避免双份账号体系

Proxy

oqbridge

负责接住 OpenSearch 风格的查询请求，基于时间范围判断应该命中热层、冷层或双路 fan-out，并在必要时合并搜索结果。

Worker

oqbridge-migrate

部署在 OpenSearch 节点附近，本地读取历史数据，使用并行 sliced scroll 抽取并压缩写入 Quickwit，尽量缩短数据搬运链路。

Capabilities

围绕真实运维场景设计的能力组合

Query Proxy

面向查询侧

  • 透明反向代理，尽量保持 OpenSearch API 使用习惯
  • 按时间范围智能路由到 OpenSearch、Quickwit 或双路并发
  • 支持 `logs-*` 等通配符索引模式
  • 跨冷热层时可合并 score 排序结果
  • 支持按索引定制时间字段与冷热阈值
Migration Worker

面向迁移侧

  • 自动解析通配符索引并过滤系统索引
  • 迁移前自动创建 Quickwit 索引，减少前置准备
  • 并行 sliced scroll 与 gzip 压缩提升迁移效率
  • 支持 checkpoint / resume，迁移中断后可继续
  • 通过分布式锁保证多实例场景下的安全协同
Production Fit

面向生产可用性

  • Quickwit 由服务账号访问，终端用户无感知
  • 迁移指标写入 OpenSearch，便于制作监控看板
  • 支持一次性任务与 cron 守护两种运行模式
  • 冷数据可配置自动保留与自动删除
  • 尽量让接入成本集中在配置，而不是改业务代码

Lifecycle

从写入、迁移到删除，一条清楚的数据时间线

Day 0

数据写入

新数据进入 OpenSearch，保持最好的近线检索体验。

Day 25

准备迁移

`oqbridge-migrate` 开始处理接近冷却边界的数据，具体天数可调。

Day 30

冷层接管

历史数据已经可从 Quickwit 查询，必要时还可以从 OpenSearch 删除副本。

Day 395

自动清理

超过 `retention.cold_days` 的冷数据由 Quickwit 自动清除，降低长期堆积压力。

Why It Matters

对团队来说，最重要的是改造成本可控

对开发者

多数情况下继续对着 OpenSearch 兼容接口发请求，不需要让业务理解底层冷热层实现细节。

对平台团队

把“存得下”和“查得快”拆开处理，让 OpenSearch 专注热数据，让 Quickwit 承担历史归档。

对运维

迁移、断点恢复、分布式锁和指标采集都已经内建，减少一堆旁路脚本和人为流程。

Quick Start

保持简单，先跑起来，再慢慢调优

构建与运行

make build
cp configs/oqbridge.yaml oqbridge.yaml

./bin/oqbridge -config oqbridge.yaml
./bin/oqbridge-migrate -config oqbridge.yaml --once

推荐部署方式

  • 在 OpenSearch 节点附近部署 `oqbridge-migrate`
  • 在客户端容易访问的位置部署 `oqbridge`
  • 共享一份配置格式，降低维护复杂度
  • 先用默认 30 天 / 365 天策略，再按索引细化
